Viacero projektov mám ako git repozitáre. V týchto projektoch používam rovnaké tools/utilieties skripty. Je možno nejak vyčleniť tieto skripty do osobitného git repa aby som po ich vylepšení v jednom projekte nemusel prenášať zmeny týchto skriptov do iných projektov?

ChatGPT said:

Áno. Najbežnejšie riešenie je vyčleniť spoločné skripty do samostatného Git repozitára, napríklad my-tools, a potom ho do ostatných projektov pripojiť jedným z týchto spôsobov.

Najlepšia praktická voľba: Git submodule

Vytvoríš samostatné repo:

Bash
tools/
  scripts/
    deploy.sh
    backup.sh
    lint.sh

Potom ho v každom projekte pridáš ako submodule:

Bash
git submodule add git@github.com:mojmir/my-tools.git tools
git commit -m "Add shared tools as submodule"

V projekte potom používaš skripty napríklad takto:

Bash
./tools/scripts/deploy.sh

Keď skripty v samostatnom repozitári vylepšíš, v inom projekte ich aktualizuješ:

Bash
cd tools
git pull
cd ..
git add tools
git commit -m "Update shared tools"

Pri klonovaní projektu treba použiť:

Bash
git clone --recurse-submodules git@github.com:mojmir/my-project.git

alebo po bežnom clone:

Bash
git submodule update --init --recursive

Alternatíva: Git subtree

git subtree je podobné riešenie, ale spoločný kód je priamo súčasťou hlavného repozitára. Výhoda je, že projekt funguje aj bez špeciálneho submodule init. Nevýhoda je, že synchronizácia so spoločným repom je trochu menej prehľadná.

Pridanie:

Bash
git subtree add --prefix=tools git@github.com:mojmir/my-tools.git main --squash

Aktualizácia:

Bash
git subtree pull --prefix=tools git@github.com:mojmir/my-tools.git main --squash

Ešte lepšie, ak sú to skripty použiteľné ako nástroj

Ak sú to napríklad Python, Node.js, PHP alebo shell nástroje, často je čistejšie spraviť z nich balík:

  • Python: vlastný package, inštalácia cez pip

  • Node.js: vlastný npm package

  • PHP: Composer package

  • Shell skripty: samostatné repo s inštalačným skriptom alebo verzované release archívy

Potom projekty závisia od konkrétnej verzie nástroja, napríklad:

Bash
pip install git+ssh://git@github.com/mojmir/my-tools.git@v1.2.0

Moje odporúčanie

Pre jednoduché zdieľané skripty by som použil Git submodule.

Je to dobré riešenie, keď chceš:

  • mať skripty v jednom centrálnom repozitári,

  • vedieť, ktorú presnú verziu používa každý projekt,

  • aktualizovať ich v projektoch iba vtedy, keď chceš,

  • nemať duplicitu medzi repozitármi.

Typická štruktúra by bola:

project-a/
  src/
  tools/       -> git submodule na shared-tools repo

project-b/
  app/
  tools/       -> git submodule na ten istý shared-tools repo

shared-tools/
  scripts/
  README.md

Dôležitá vec: submodule sa neaktualizuje úplne automaticky vo všetkých projektoch. Každý projekt si pamätá konkrétny commit spoločného repa. To je väčšinou výhoda, lebo zmena v skriptoch ti nerozbije všetky projekty naraz.

Odporúčaná jednoduchá štruktúra bez src/

Toto je úplne v poriadku:

my-tools/
  pyproject.toml
  my_tools/
    __init__.py
    docker_info.py
    git_cleanup.py

Toto sa volá flat layout. Setuptools ho podporuje: package adresár je priamo pod koreňom projektu vedľa pyproject.toml. Dokumentácia však upozorňuje, že flat layout môže byť pri väčších projektoch náchylnejší na omyly, napríklad pri testoch alebo keď máš v root adresári veľa ďalších Python súborov. setuptools.pypa.io

pyproject.toml:

TOML
[build-system]
requires = ["setuptools>=77"]
build-backend = "setuptools.build_meta"

[project]
name = "my-tools"
version = "1.0.0"
requires-python = ">=3.10"

[project.scripts]
docker-info = "my_tools.docker_info:main"
git-cleanup = "my_tools.git_cleanup:main"

Všimni si rozdiel:

TOML
name = "my-tools"

To je názov balíka pre pip.

Ale v Pythone importuješ:

Python
import my_tools

To je názov Python modulu/package.

Môžeš mať súbory aj priamo v root adresári?

Áno, ale musel by si to spraviť skôr takto:

my-tools/
  pyproject.toml
  docker_info.py
  git_cleanup.py

Bez __init__.py.

A v pyproject.toml by si explicitne uviedol top-level moduly:

TOML
[build-system]
requires = ["setuptools>=77"]
build-backend = "setuptools.build_meta"

[project]
name = "my-tools"
version = "1.0.0"
requires-python = ">=3.10"

[tool.setuptools]
py-modules = ["docker_info", "git_cleanup"]

[project.scripts]
docker-info = "docker_info:main"
git-cleanup = "git_cleanup:main"

Setuptools má na toto nastavenie py-modules a zároveň odporúča používať [project.scripts] pre konzolové príkazy. setuptools.pypa.io

Tento variant je použiteľný, ale menej čistý. Nevýhody:

  • moduly docker_info a git_cleanup budú globálne top-level názvy,

  • ľahšie vznikne konflikt s inými modulmi,

  • pri viacerých súboroch sa to časom horšie udržiava,

  • keď neskôr pridáš spoločné pomocné funkcie, budeš to pravdepodobne aj tak presúvať do package adresára.

Prečo som predtým ukázal src/

src/ layout je robustnejší, hlavne pre knižnice a väčšie balíky:

my-tools/
  pyproject.toml
  src/
    my_tools/
      __init__.py
      docker_info.py
      git_cleanup.py

Pri src/ layoute je menšia šanca, že počas vývoja omylom importuješ súbor priamo z pracovného adresára namiesto nainštalovaného balíka. Setuptools pri flat layoute tiež upozorňuje, že automatická discovery môže odmietnuť build, ak nájde viacero top-level balíkov alebo modulov, aby sa omylom nezabalili súbory, ktoré nemali byť distribuované. setuptools.pypa.io

Moje odporúčanie pre tvoje utility skripty

Použi toto:

my-tools/
  pyproject.toml
  my_tools/
    __init__.py
    docker_info.py
    git_cleanup.py

Je to jednoduchšie než src/, ale stále dosť čisté.

Každý skript uprav takto:

Python

def main():
    print("Docker info...")


if __name__ == "__main__":
    main()

A potom ho po inštalácii spustíš:

Bash

Ale stále ho vieš spustiť aj priamo počas vývoja:

Bash
python -m my_tools.docker_info

Teda: áno, nemusíš použiť src/, ale namiesto __init__.py v root adresári radšej vytvor adresár my_tools/.

Ešte k verziám (python -m pip install "git+ssh://git@github.com/mojmir/my-tools.git@v1.3.0") - ak by som ich nechcel riešiť - jednoducho vždy budeme chceť nainštalovať skripty tak ako sú a ak by niekto preinštalovával tak si chce jednoducho stiahnuť novši stav (naschval nehovorím "verziu") skriptu. Ako toto dosiathnúť?

ChatGPT said:

Toto sa dá nastaviť jednoducho: namiesto tagu použiješ vetvu, typicky main, a pri aktualizácii vynútiš reinštaláciu. Dôležitý detail je, že samotné --upgrade nemusí stačiť, ak v balíku nemeníš version.

Áno. Použi vetvu namiesto tagu:

Bash
python -m pip install "my-tools @ git+ssh://git@github.com/mojmir/my-tools.git@main"

A keď chceš stiahnuť najnovší stav z main, daj:

Bash
python -m pip install --upgrade --force-reinstall "my-tools @ git+ssh://git@github.com/mojmir/my-tools.git@main"

pip pri Git URL podporuje za @ nielen tag, ale aj branch name, commit hash alebo iný Git ref, takže @main znamená “ber aktuálny stav vetvy main”. pip.pypa.io

Dôležité je práve toto:

Bash
--force-reinstall

Lebo keď nebudeš meniť version v pyproject.toml, obyčajné --upgrade nemusí byť dosť intuitívne. pip štandardne necháva nainštalovanú verziu tak, ako je, pokiaľ nemá dôvod ju meniť; --force-reinstall hovorí, že má balík preinštalovať aj vtedy, keď sa javí ako už aktuálny. pip.pypa.io

V pyproject.toml si pokojne nechaj technickú verziu stále rovnakú, napríklad:

TOML
[project]
name = "my-tools"
version = "0.0.0"

A mentálne ju vôbec nerieš. Potom “aktuálnosť” nebudeš riadiť cez verzie, ale cez stav vetvy main.

Praktický variant

Sprav si v každom projekte napríklad súbor:

Bash
# update-tools.sh
python -m pip install --upgrade --force-reinstall \
  "my-tools @ git+ssh://git@github.com/mojmir/my-tools.git@main"

Potom aktualizácia nástrojov bude len:

Bash
./update-tools.sh

Variant cez requirements súbor

Môžeš mať napríklad requirements-tools.txt:

my-tools @ git+ssh://git@github.com/mojmir/my-tools.git@main

Prvé nainštalovanie:

Bash
python -m pip install -r requirements-tools.txt

Aktualizácia na najnovší stav:

Bash
python -m pip install --upgrade --force-reinstall -r requirements-tools.txt

Len pozor: ak dáš --force-reinstall -r requirements.txt na veľký requirements súbor, môže ti to preinštalovať aj ostatné balíky. Preto by som pre tieto tvoje utility skripty mal radšej samostatný requirements-tools.txt alebo jednoduchý update-tools.sh.

Moje odporúčanie

Pre tvoj prípad by som používal toto:

Bash
python -m pip install --upgrade --force-reinstall \
  "my-tools @ git+ssh://git@github.com/mojmir/my-tools.git@main"

Tým dosiahneš: “nainštaluj mi aktuálny stav skriptov z hlavnej vetvy”, bez riešenia tagov typu v1.3.0.
